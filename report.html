Deterministic · Repeatable

Traditional Vulnerabilities

These vulnerabilities produce the same exploit result every time. Same input = same exploit. Detectable with standard SAST tools and verifiable with deterministic tests.

The JWT signing secret "do_not_share_this" was hardcoded in source code. Anyone with repo access can forge valid tokens for any user.

Detected by: Semgrep (jwt-python-hardcoded-secret)

Vulnerable
SECRET = "do_not_share_this"
Fixed
SECRET = os.environ.get("JWT_SECRET",
    "fallback-dev-secret-change-in-production")

Django's SECRET_KEY was hardcoded with the django-insecure- prefix. Enables session hijacking, CSRF bypass, and cookie tampering.

Detected by: Manual Review

Vulnerable
SECRET_KEY = "django-insecure-^3!*dt2xq2vs7..."
Fixed
SECRET_KEY = os.environ.get("DJANGO_SECRET_KEY",
    "fallback-only-for-dev-not-for-production")

Django's |safe template filter was used on user-controlled notes and content fields, rendering raw HTML including <script> tags in the victim's browser.

Detected by: Manual Review

Vulnerable
{{ archive.notes|safe }}
{{ archive.content|safe }}
Fixed
{{ archive.notes }}
{{ archive.content }}

User input was interpolated directly into a raw SQL query via f-string. Allowed UNION-based attacks to extract password hashes and cross-user data.

Detected by: Semgrep (sql-injection-db-cursor-execute, tainted-sql-string, formatted-sql-query, sqlalchemy-execute-raw-query)

Vulnerable
sql = f"SELECT ... WHERE title LIKE '%{query}%'"
cursor.execute(sql)
Fixed
archives = Archive.objects.filter(
    user=request.user,
    title__icontains=query
).select_related("user")

View, edit, and delete endpoints only checked if an archive exists — not if it belonged to the requesting user. Any authenticated user could access any archive by ID.

Detected by: Manual Review

Vulnerable
archive = get_object_or_404(Archive, pk=archive_id)
Fixed
archive = get_object_or_404(Archive, pk=archive_id,
    user=request.user)

Debug mode exposes full stack traces, settings values, database paths, and installed apps on any error page.

Vulnerable
DEBUG = True
Fixed
DEBUG = os.environ.get("DEBUG", "False").lower() == "true"

Empty ALLOWED_HOSTS is permissive when DEBUG=True and rejects all requests when DEBUG=False.

Vulnerable
ALLOWED_HOSTS = []
Fixed
ALLOWED_HOSTS = os.environ.get(
    "ALLOWED_HOSTS", "localhost,127.0.0.1"
).split(",")

The "add archive" endpoint fetched any user-supplied URL without validation. Attackers could probe internal services, steal cloud metadata credentials (AWS 169.254.169.254), and scan internal networks.

Detected by: Semgrep (ssrf-injection-requests)

Vulnerable
response = requests.get(url, timeout=10)
# No URL validation at all
Fixed
if not _is_url_safe(url):
    messages.error(request, "Blocked URL")
    return render(...)
# Validates scheme, resolves DNS,
# blocks private/reserved IPs

Raw SQL errors were displayed to users, revealing table names, column counts, and query structure — helping attackers refine injection payloads.

Vulnerable
messages.error(request, f"SQL Error: {str(e)}")
Fixed
messages.error(request,
    "An error occurred while searching.")

Probabilistic · LLM-Dependent

LLM / AI Vulnerabilities

These vulnerabilities depend on non-deterministic LLM behavior. The same prompt may produce different results across runs. Tests use mocked LLM responses for deterministic proof, while the code-level guardrails protect against any LLM output.

Key Distinction: Traditional vulnerabilities are fixed by removing the vulnerable code pattern. LLM vulnerabilities are fixed by adding guardrails around what the LLM output can do — the LLM itself is untrusted.

Attack Vector

Hidden HTML in archive content (display:none) instructs the LLM to use the fetch_url tool to send user notes (containing PII like passwords and SSNs) to an attacker-controlled URL.

Three-Layer Fix

1. Strip hidden HTML before passing content to LLM — removes display:none elements and HTML comments.

2. Don't pass user notes to the LLM prompt — the sensitive data never reaches the model.

3. Domain allowlist for tool call URLs — only allows URLs matching the archive's original domain or explicitly provided by the user.

Vulnerable
prompt = f"""
  Archive Content: {archive.content}
  Archive Notes: {archive.notes}
"""
# Tool calls executed blindly:
requests.get(url_to_fetch, timeout=5)
Fixed
cleaned = _strip_hidden_html(archive.content)
prompt = f"""
  Archive Content: {cleaned}
"""
# Notes removed from prompt entirely
# Tool URLs checked against domain allowlist
if fetch_domain not in allowed_domains:
    # BLOCKED

Attack Vector

The LLM was asked to generate a file path for exported summaries. A manipulated prompt could cause it to return paths like /etc/passwd or ../../../app/manage.py, and the server would write to that location.

Vulnerable
file_path = query_llm(path_prompt).strip()
with open(file_path, "w") as f:
    f.write(summary_content)
# LLM controls the file path!
Fixed
safe_filename = re.sub(r"[^a-zA-Z0-9._-]", "_", hint)
file_path = os.path.join(base_dir, safe_filename)
file_path = os.path.realpath(file_path)
if not file_path.startswith(base_dir):
    # BLOCKED - path traversal attempt

Attack Vector

Natural language queries were converted to SQL by the LLM, then executed directly against the database with no validation. Prompt injection could produce DELETE, DROP TABLE, or cross-user SELECT queries.

Vulnerable
sql_query = query_llm(user_input, ...)
cursor.execute(sql_query)
# Executes ANYTHING the LLM generates
Fixed
if not normalized.upper().startswith("SELECT"):
    answer = "Only SELECT queries are allowed."
else:
    # Force user_id filter
    enforced = query + f" AND user_id = {user.id}"
    cursor.execute(enforced)
